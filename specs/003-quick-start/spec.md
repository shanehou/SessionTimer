# Feature Specification: 快速开始与预备倒计时

**Feature Branch**: `003-quick-start`  
**Created**: 2026-03-07  
**Status**: Draft  
**Input**: User description: "有时候用户只是想先试试看，而不是先想好了再用。所以我想做一个快速开始的单独功能，可以是一个单独的页面，用户直接在页面上设定项目、组数、间隔等信息就可以直接开始了，结束后可以弹窗提示用户是否保存为计划。这样用户可以更低门槛地使用。另外，用户点击开始后立刻倒计时不符合实际情况，因为用户点击屏幕后还要做一些准备，所以我希望能够让用户设定一个预准备时间。"

## Clarifications

### Session 2026-03-07

- Q: 预备倒计时的默认值来源——是否有全局默认值自动填入，还是每次独立设置？ → A: 纯Session级别，用户每次在快速开始页面和Session编辑中独立设置，无全局默认
- Q: 快速开始页面再次打开时是否记忆上次配置？ → A: 记忆上次配置，自动填入上一次使用的配置，但App重启后丢失（仅内存缓存）
- Q: 保存提示的触发条件——仅100%完成时提示，还是中途结束也提示？ → A: 始终提示，无论训练完成度如何，用户结束快速开始模式时都提示是否保存配置
- Q: 预备倒计时（Preparing）阶段的全屏背景色主题？ → A: 蓝底白字（与Work黑底、Rest绿底形成三色区分）

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 快速配置并立即开始计时 (Priority: P1)

用户不想先创建和保存一个完整的Session计划，而是想直接打开一个专用的"快速开始"页面，在页面上快速设定练习项目名称、组数、每组练习时长、休息时长，然后立刻开始计时。整个流程追求最低门槛、最少步骤，让用户"想练就练"。

**Why this priority**: 这是本功能的核心价值——降低使用门槛。如果用户每次都必须先创建并保存Session才能计时，会劝退那些只想"先试试看"的用户。快速开始页面让用户无需任何前置操作即可开始训练。

**Independent Test**: 打开快速开始页面，设定一个项目（如"俯卧撑"，3组，30秒练习，15秒休息），点击开始，验证计时器能正确按设定运行。

**Acceptance Scenarios**:

1. **Given** 用户在App主界面，**When** 用户点击"快速开始"入口，**Then** 系统显示快速开始页面，页面包含项目配置区域和开始按钮
2. **Given** 用户在快速开始页面，**When** 用户设定项目名称、组数、练习时长、休息时长，**Then** 系统实时展示配置摘要，用户可以确认设置
3. **Given** 用户已完成至少一个项目的配置，**When** 用户点击开始按钮，**Then** 系统立即进入计时界面并开始执行（如设置了预备时间则先进行预备倒计时）
4. **Given** 用户在快速开始页面，**When** 用户未设定任何项目就点击开始，**Then** 系统提示用户至少添加一个练习项目

---

### User Story 2 - 添加多个练习项目 (Priority: P2)

用户在快速开始页面不仅可以设定一个练习项目，还可以添加多个项目组成一个完整的临时Session。例如健身时可能先做深蹲再做卧推，用户可以在快速开始页面依次添加这些项目。

**Why this priority**: 多项目支持让快速开始功能覆盖更多真实训练场景，但即使只支持单个项目，快速开始的核心价值（低门槛）已经成立。

**Independent Test**: 在快速开始页面添加3个不同项目，分别设定不同的组数和时长，点击开始后验证计时器按顺序执行所有项目。

**Acceptance Scenarios**:

1. **Given** 用户在快速开始页面已配置了一个项目，**When** 用户点击"添加项目"，**Then** 页面新增一个项目配置区域
2. **Given** 用户添加了多个项目，**When** 用户查看快速开始页面，**Then** 所有项目按添加顺序排列，用户可以清晰看到每个项目的设置
3. **Given** 用户添加了多个项目，**When** 用户想要移除某个项目，**Then** 用户可以删除该项目，且其余项目保持不变
4. **Given** 用户添加了多个项目并点击开始，**When** 第一个项目的所有组完成，**Then** 系统自动进入下一个项目的计时

---

### User Story 3 - 完成后保存为计划 (Priority: P3)

用户通过快速开始完成训练后，系统弹窗询问是否将本次配置保存为一个正式的Session计划，方便下次直接复用。这样用户可以先体验、再决定是否保留。

**Why this priority**: 保存功能打通了"试用"到"常用"的路径，让快速开始产出的配置不被浪费。但即使没有保存功能，用户也已获得了核心的低门槛计时体验。

**Independent Test**: 通过快速开始完成一次训练，在弹窗中选择保存，验证保存后的Session出现在主界面的Session列表中，且可以正常启动。

**Acceptance Scenarios**:

1. **Given** 用户通过快速开始结束了训练（自然完成或手动结束），**When** 系统显示结束界面，**Then** 弹窗询问用户"是否将本次训练配置保存为计划？"
2. **Given** 用户在保存弹窗中选择"保存"，**When** 用户输入Session名称（默认可自动生成），**Then** 系统将快速开始的所有配置保存为一个正式的Session，出现在主界面Session列表中
3. **Given** 用户在保存弹窗中选择"不保存"，**When** 弹窗关闭，**Then** 系统返回主界面，本次配置不被持久化
4. **Given** 用户之前通过快速开始保存了一个Session，**When** 用户在主界面点击该Session，**Then** 该Session可以像正常创建的Session一样使用（查看、编辑、启动）

---

### User Story 4 - 预备倒计时 (Priority: P4)

用户点击"开始"后，计时器不立刻进入正式练习倒计时，而是先执行一段用户可配置的"预备时间"倒计时。这段时间让用户从点击屏幕到真正就位（如握住器械、拿起乐器、摆好姿势）有充足的准备。

**Why this priority**: 预备倒计时解决了一个真实的使用痛点——用户点击开始后手忙脚乱地准备。但这是一个增强体验的功能，不影响核心的计时流程。

**Independent Test**: 设定5秒预备时间，启动计时，验证系统先显示5秒预备倒计时，倒计时结束后自动进入第一个项目的Work阶段。

**Acceptance Scenarios**:

1. **Given** 用户在快速开始页面或Session设置中设定了预备时间（如5秒），**When** 用户点击开始，**Then** 系统先显示预备倒计时界面，显示"准备"字样和倒计时秒数
2. **Given** 预备倒计时正在进行，**When** 倒计时归零，**Then** 系统自动切换到第一个项目的Work阶段，开始正式计时
3. **Given** 预备倒计时正在进行，**When** 用户想跳过预备时间（双击屏幕），**Then** 系统立即结束预备倒计时，进入正式计时
4. **Given** 用户未设定预备时间（设为0秒），**When** 用户点击开始，**Then** 系统直接进入第一个项目的Work阶段，与现有行为一致
5. **Given** 用户从主界面启动一个已保存的Session，**When** 该Session设置了预备时间，**Then** 系统同样先执行预备倒计时再进入正式计时

---

### Edge Cases

- 用户在快速开始页面设定了项目后，误操作退出页面：系统不保存配置（临时配置不持久化），但可考虑短期缓存以支持恢复
- 用户设定练习时长或休息时长为0秒：系统应允许，0秒的阶段将被自动跳过
- 用户在快速开始页面添加大量项目（如超过20个）：系统应正常处理，页面应支持滚动
- 预备倒计时过程中用户暂停（单击）：预备倒计时暂停，再次单击继续
- 用户在预备倒计时过程中长按结束：系统结束整个Session，返回快速开始页面或主界面
- 用户通过快速开始计时中途手动结束（长按结束）：仍提示是否保存配置，因为配置本身与执行进度无关
- 快速开始的训练过程中App进入后台：行为与已保存Session的后台计时保持一致

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须提供一个独立的"快速开始"入口，用户可以从主界面直接访问
- **FR-002**: 快速开始页面必须允许用户配置至少一个练习项目，每个项目包含：项目名称（可选，有默认值）、组数、每组练习时长、每组休息时长
- **FR-003**: 快速开始页面必须允许用户添加多个练习项目，支持项目的增删和排序
- **FR-004**: 用户在快速开始页面点击开始后，系统必须直接进入计时执行流程，无需保存步骤
- **FR-005**: 快速开始模式的计时执行过程必须与已保存Session的计时体验完全一致（手势控制、感官反馈、后台运行、Live Activities等）
- **FR-006**: 快速开始模式结束时（无论是自然完成还是用户手动结束），系统必须通过弹窗询问用户是否将配置保存为正式Session
- **FR-007**: 用户选择保存时，系统必须将快速开始的配置转化为标准Session数据结构并持久化，保存后的Session与手动创建的Session无差异
- **FR-008**: 用户选择不保存时，系统必须丢弃临时配置数据
- **FR-009**: 系统必须支持用户可配置的预备倒计时时间，适用于快速开始和已保存Session
- **FR-010**: 预备倒计时期间，系统必须在屏幕上显示明确的"准备"状态标识和倒计时秒数，使用蓝底白字全屏背景（与Work黑底、Rest绿底形成三色区分）
- **FR-011**: 预备倒计时结束后，系统必须自动切换到第一个项目的Work阶段
- **FR-012**: 预备倒计时时间设为0时，系统必须跳过预备阶段，直接开始计时（兼容现有行为）
- **FR-013**: 预备倒计时期间必须支持现有的手势控制（单击暂停/继续、双击跳过、长按结束）
- **FR-014**: 快速开始页面上的项目配置字段必须提供合理的默认值，减少用户输入量
- **FR-015**: 快速开始页面再次打开时，必须自动填入用户上一次使用的配置（项目列表及各项参数）；App重启后该缓存丢失，恢复为默认值

### Key Entities

- **快速开始配置（临时）**: 用户在快速开始页面创建的临时训练配置，数据结构与Session一致但未持久化。包含一个或多个临时Block（项目名称、组数、练习时长、休息时长），在用户选择保存后转化为正式Session。
- **预备时间设置**: 用户可配置的预备倒计时时长（单位：秒），在快速开始页面和Session编辑中独立设置，无全局默认值。每次新建快速开始或新Session时，预备时间默认为0秒（不启用预备倒计时）。
- **Timer State（扩展）**: 在现有计时状态基础上增加"预备（Preparing）"阶段，作为Work/Rest之外的第三种计时阶段。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从打开快速开始页面到开始第一次计时的操作时间不超过30秒（单项目场景）
- **SC-002**: 90%的首次使用用户能在无引导的情况下成功通过快速开始完成一次训练
- **SC-003**: 快速开始保存为Session后，保存的Session与手动创建的Session在功能上无差异
- **SC-004**: 预备倒计时的时间精度偏差不超过1秒
- **SC-005**: 用户通过快速开始启动训练的步骤数不超过3步（打开页面 → 配置项目 → 点击开始）

## Assumptions

- 快速开始的临时配置数据在用户未保存的情况下不做磁盘持久化，但在App运行期间保留内存缓存，再次打开快速开始页面时自动填入上次配置；App重启后缓存丢失，恢复默认值
- 快速开始页面的项目配置默认值为：项目名称"项目 1/2/3..."、组数3、练习时长30秒、休息时长15秒
- 预备倒计时是Session级别的独立设置，无全局默认值；每次新建快速开始或新Session时默认为0秒
- 预备倒计时的可选时长范围为0秒至30秒
- 快速开始模式的计时界面复用现有的计时器界面，不需要单独设计
- 保存弹窗中的Session默认名称可基于日期和时间自动生成（如"快速训练 2026-03-07 14:30"）
