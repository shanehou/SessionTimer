# Feature Specification: 本地 TTS 模型替换

**Feature Branch**: `004-local-tts-model`
**Created**: 2026-03-07
**Status**: Draft
**Input**: User description: "我想让语音更自然一些，不使用自带的TTS，使用本地单独的部署的模型。"

## Clarifications

### Session 2026-03-07

- Q: vendor 目录中包含 5 个模型变体（steps-2 到 steps-6），应使用哪个？ → A: 使用 model-steps-6（最高质量变体），优先语音自然度
- Q: 模型文件（约 130MB）如何分发给用户？ → A: 直接打包到应用中，保证开箱即用，无需网络下载
- Q: 语音合成的时机？ → A: 用户保存时预生成音频文件并缓存，计时执行时直接播放缓存的音频文件；默认播报文本（"准备"、"休息"、"继续"、"练习完成"）的音频预先内置在应用中

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 使用本地模型生成的高质量语音进行播报 (Priority: P1)

用户在使用 Session Timer 进行训练时，所有语音播报使用本地 TTS 模型预先生成的音频文件播放，替代当前的系统自带 TTS（AVSpeechSynthesizer）。用户应感受到语音更加自然、流畅，接近真人发声效果。

语音音频文件在用户保存 Block 或 Session 设置时由本地模型生成并缓存，计时执行时直接播放缓存的音频文件，无推理延迟。对于默认播报文本（"准备"、"休息"、"继续"、"练习完成"），应用内置预生成的音频文件，用户无需等待生成即可使用。

播报的触发规则和播报内容保持不变（Block 首组 Work 播报开始内容、Rest 播报休息内容、后续 Work 播报继续内容、Session 完成播报完成内容），仅语音合成引擎和播放方式发生改变。

**Why this priority**: 高质量语音播报是本功能的核心目标。预生成 + 缓存播放的架构确保了语音质量提升的同时不引入任何运行时延迟。

**Independent Test**: 创建一个包含多个 Block 的 Session 并保存，启动计时，验证所有阶段切换时的语音播报音质自然、发音准确、无延迟。

**Acceptance Scenarios**:

1. **Given** 用户使用默认播报文本启动一个 Session，**When** 阶段切换触发语音播报，**Then** 播报使用应用内置的预生成音频，语音自然清晰
2. **Given** 用户为 Block 设置了自定义播报文本并保存，**When** 该 Block 的对应阶段开始，**Then** 播报使用保存时生成的缓存音频文件
3. **Given** 播报文本为中文（如"深蹲"），**When** 语音播报触发，**Then** 中文发音自然准确，无明显机器感
4. **Given** 播报文本为英文（如"Squat"），**When** 语音播报触发，**Then** 英文发音自然准确
5. **Given** 播报文本为中英混合（如"C大调 Scale"），**When** 语音播报触发，**Then** 混合内容能被合理朗读
6. **Given** 设备处于离线状态，**When** 阶段切换触发语音播报，**Then** 语音播报正常工作（音频文件已提前缓存在本地）

---

### User Story 2 - 保存时生成语音并缓存 (Priority: P2)

用户在编辑 Block 或 Session 的播报文本后点击保存时，系统在后台使用本地 TTS 模型为该文本生成音频文件并缓存。生成过程不阻塞保存操作，用户可以继续使用应用。当 Block 名称变更时（名称作为默认的开始播报内容），也需要重新生成对应的音频。

**Why this priority**: 预生成是保证播放时零延迟的前提。没有预生成机制，就无法实现即时播放。

**Independent Test**: 编辑 Block 的播报文本并保存，验证音频文件被生成和缓存；再次启动计时验证播放的是缓存的音频。

**Acceptance Scenarios**:

1. **Given** 用户修改了 Block 的自定义播报文本，**When** 用户保存设置，**Then** 系统在后台为新文本生成音频文件并缓存，保存操作不被阻塞
2. **Given** 用户修改了 Block 名称（未设置自定义开始播报文本），**When** 用户保存设置，**Then** 系统为新的 Block 名称生成音频文件并缓存
3. **Given** 用户修改了 Session 的完成播报文本，**When** 用户保存设置，**Then** 系统为新的完成文本生成音频文件并缓存
4. **Given** 音频生成尚未完成，**When** 用户启动计时且需要播放该文本，**Then** 系统降级使用 AVSpeechSynthesizer 实时播报

---

### User Story 3 - 内置默认语音与后台播放兼容 (Priority: P3)

应用内置默认播报文本（"准备"、"休息"、"继续"、"练习完成"）的预生成音频文件，用户安装后不需要任何生成步骤即可获得高质量语音播报。同时，语音播报在后台和音乐混音场景下应正常工作。

**Why this priority**: 内置默认音频保证了开箱即用的体验，用户无需做任何设置就能感受到语音质量的提升。后台兼容性是已有功能的延续。

**Independent Test**: 安装应用后直接使用默认设置启动 Session，验证所有默认播报使用内置的高质量音频；切到后台并播放音乐，验证播报正常。

**Acceptance Scenarios**:

1. **Given** 用户首次安装应用，未做任何自定义设置，**When** 启动 Session 后阶段切换发生，**Then** 默认播报文本使用内置的高质量预生成音频播放
2. **Given** 用户清空了自定义播报文本（回退到默认值），**When** 对应阶段开始，**Then** 使用内置的默认音频播放
3. **Given** 应用在后台运行，**When** 阶段切换发生，**Then** 语音播报正常播放
4. **Given** 用户正在播放音乐，**When** 语音播报触发，**Then** 语音与音乐混音播放，播报期间适当压低音乐音量
5. **Given** 手机处于静音模式，**When** 阶段切换发生，**Then** 语音播报行为与现有音效行为一致

---

### Edge Cases

- 本地模型文件缺失或损坏时：所有语音生成降级为 AVSpeechSynthesizer，保证播报功能不中断
- 缓存的音频文件被系统清理时：下次播放时发现缓存缺失，降级使用 AVSpeechSynthesizer；后台重新触发音频生成
- 播报文本包含特殊字符或纯数字时：模型应能合理处理，生成失败时降级到 AVSpeechSynthesizer
- 用户快速多次修改播报文本并保存时：取消之前未完成的生成任务，只为最终保存的文本生成音频
- 设备存储空间不足时：音频文件生成失败，降级使用 AVSpeechSynthesizer 实时播报
- 阶段快速切换时（Work/Rest 时长很短）：当前音频播放被中断，立即开始新的音频播放
- Block 名称变更但未显式设置开始播报文本时：需要为新名称重新生成音频

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须使用本地部署的 TTS 模型替代 AVSpeechSynthesizer 进行语音合成
- **FR-002**: 本地 TTS 模型必须完全离线运行，不依赖网络连接
- **FR-003**: 本地 TTS 模型文件必须直接打包在应用安装包内，用户安装后即可使用，无需额外下载
- **FR-004**: 本地 TTS 模型必须支持中文和英文文本的语音合成
- **FR-005**: 应用必须内置默认播报文本（"准备"、"休息"、"继续"、"练习完成"）的预生成音频文件，安装后即可使用
- **FR-006**: 用户保存 Block 或 Session 设置时，系统必须在后台为自定义播报文本生成音频文件并缓存到本地
- **FR-007**: 用户修改 Block 名称时（名称作为默认开始播报内容），系统必须为新名称生成音频文件并缓存
- **FR-008**: 音频生成必须在后台执行，不阻塞用户的保存操作和界面交互
- **FR-009**: 计时执行时，系统必须直接播放缓存的音频文件，不进行实时语音合成
- **FR-010**: 当缓存的音频文件不可用时（未生成完成、文件丢失等），系统必须降级使用 AVSpeechSynthesizer 实时播报
- **FR-011**: 当新的播报请求到来时，系统必须能够中断当前正在播放的音频并立即播放新的音频
- **FR-012**: 音频播放必须通过现有的音频会话进行，保持与音乐混音和后台播放的兼容性
- **FR-013**: 所有现有的播报触发规则、播报内容逻辑和自定义文本功能必须保持不变
- **FR-014**: 语音播报开关的行为必须保持不变

### Key Entities

- **SpeechService（重构）**: 现有的语音服务需要重构为"预生成 + 缓存播放"架构：保存时调用本地模型生成音频并缓存，计时时播放缓存音频，同时保留 AVSpeechSynthesizer 作为降级方案
- **本地 TTS 模型**: 已准备在 vendor 目录中的本地 TTS 模型资源（sherpa-onnx 框架 + matcha-icefall 模型文件），用于生成音频文件
- **音频缓存**: 存储预生成的音频文件，以播报文本为标识进行管理；内置默认音频随应用安装，自定义音频在用户保存时生成
- **Block（扩展影响）**: Block 名称变更或自定义播报文本变更时触发音频重新生成
- **Session（扩展影响）**: 完成播报文本变更时触发音频重新生成

## Assumptions

- 本地 TTS 模型文件已准备就绪（vendor 目录中的 sherpa-onnx.xcframework 和 matcha-icefall-zh-en 模型），使用 model-steps-6 变体（最高质量）
- 应用仅打包 model-steps-6.onnx 及 vocos-16khz-univ.onnx，不包含其他 steps 变体，以控制包体积
- 默认播报音频文件（"准备"、"休息"、"继续"、"练习完成"）在开发阶段预先生成并作为应用资源内置
- 缓存的音频文件体积较小（短文本语音，每个文件预计几十 KB），对设备存储影响可忽略
- 本地模型的中英文语音质量明显优于系统自带的 AVSpeechSynthesizer
- 播报触发逻辑、自定义文本、语音开关等现有功能保持不变，本次改变的是语音合成时机（从实时合成改为预生成）和合成引擎
- 音频生成在后台线程异步执行，不影响用户操作流畅度

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户在语音播报的自然度评估中，本地模型生成的语音明显优于系统自带 TTS，听感接近真人发声
- **SC-002**: 计时执行时语音播报即时播放，无可感知的合成延迟（音频已预先缓存）
- **SC-003**: 用户保存设置时，音频生成在后台完成，保存操作无卡顿感
- **SC-004**: 所有现有的播报功能（四种场景播报、自定义文本、语音开关、后台播报、音乐混音）在引擎替换后继续正常工作
- **SC-005**: 缓存的音频不可用时，系统自动降级到系统 TTS，用户仍能获得语音播报服务
- **SC-006**: 用户首次安装应用后，使用默认设置即可获得高质量语音播报（内置默认音频）
